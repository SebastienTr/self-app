### Phase 2b: Technical Problems Deep Dive

**Interactive Focus:** Confronting ideas with real technical constraints — API connections, auth, data freshness, rendering, offline, performance.

**Key Breakthroughs:**

- The HTTP generic connector is the Swiss army knife — covers 80% of cases
- The agent abstracts ALL technical complexity from non-technical users
- Rendering solved by pre-compiled components + JSON config (not code generation)
- Offline-first with graceful degradation (3 layers)
- Performance via dual-loop architecture (fast render + slow agent in background)

**Ideas Generated:**

**Data & Connectivity Problems:**
- Problem #1: Progressive Elicitation (funnel conversation, one question at a time)
- Problem #2: Connector Catalog (community-maintained, like plugins)
- Problem #3: Auth Hell Solved (3 levels: API key simple → OAuth flow → custom)
- Problem #4: Secure Credential Storage (Keychain/Keystore, biometric gate)
- Problem #5: Non-API Businesses (manual entry + OCR photo + spreadsheet as backend)
- Problem #6: Data Freshness & Cost (smart polling based on user behavior)
- Problem #7: Generic Connection Framework (abstraction layer: brick → connector → auth → secure storage)
- Problem #8: Non-Technical User Configures API (zero jargon, agent translates intent)
- Problem #9: API Changes/Dies (community connector maintenance + immune system)

**Dynamic Rendering:**
- Render #1: Pre-Compiled Component System (agent generates JSON config, not code)
- Render #2: Three Rendering Levels (L1=config 90%, L2=composed template 9%, L3=generated 1%)
- Render #3: Brick DSL — declarative language like Terraform for mobile UI
- Render #4: Preview Before Commit (canary deployment for UI)

**Offline:**
- Offline #1: Offline-First 3-Layer Architecture (always available → degraded intelligence → sync on return)
- Offline #2: Intelligent Pre-Fetch (agent anticipates offline — pre-loads before sailing trip)
- Offline #3: Offline Logbook (local model handles basics, queues everything for sync)
- Offline #4: Contextual Local Model (specialized small model based on current activity)

**Performance:**
- Perf #1: Dual Loop — Fast (< 100ms, cached render, zero LLM) / Slow (1-30s, agent thinking, background)
- Perf #2: Pre-Compute via Crons (morning briefing prepared at 7am, ready when user opens at 7:15)
- Perf #3: Multi-Level Cache (L1 RAM → L2 local storage → L3 agent cache → L4 network)
- Perf #4: Intelligent Model Routing (80% of interactions = zero LLM calls)
- Perf #5: Progressive Rendering (brick by brick, fastest cache first)
- Perf #6: Battery as Biological Constraint (throttle when low, prioritize vital functions)

### Phase 3b: Memory System Deep Dive

**Interactive Focus:** Research-backed exploration of AI agent memory systems, comparing OpenClaw, Mem0, Letta, Zep, and Claude Code to design Self's memory architecture.

**Key Breakthroughs:**

- "Virtual memory for cognition" pattern (LLM context = RAM, external storage = disk)
- Mem0's ADD/UPDATE/DELETE/NOOP pipeline is the anti-repetition solution
- OpenClaw proves Markdown + SQLite is sufficient for excellent memory
- Agentic memory (agent manages its own memory) is the 2026 trend surpassing RAG

**Ideas Generated:**
- Memory #1: 4-Layer Architecture (SOUL always in context → Core Memory always in context → Semantic Store on-demand → Episodic Store on-demand)
- Memory #2: Anti-Repetition Pipeline inspired by Mem0 (ADD/UPDATE/DELETE/NOOP via lightweight LLM)
- Memory #3: Preventive Compaction inspired by OpenClaw (save important notes THEN compress old messages)
- Memory #4: User Relationship Graph (Seb → owns → sailboat → needs → marine weather)
- Memory #5: Bi-temporal Memory inspired by Zep (event time + ingestion time, supersede don't delete)
- Memory #6: Agent Manages Own Memory via tools (memory_save, memory_search, memory_update, memory_rethink)
- Memory #7: Contextual Pre-Load at Wake (Sunday morning Brittany ≠ Monday morning office)
- Memory #8: Local-First Memory (SQLite + sqlite-vec on device, Markdown readable, privacy-first)

**Memory Architecture Summary:**
```
SOUL.md + CORE MEMORY (always in context, ~2500 tokens)
    ↓
ANTI-REPETITION PIPELINE (ADD/UPDATE/DELETE/NOOP per interaction)
    ↓
SEMANTIC STORE + EPISODIC STORE + RELATIONSHIP GRAPH
(SQLite + vectors + BM25 + temporal decay)
    ↓
HYBRID SEARCH (0.7 × cosine + 0.3 × BM25 + graph + decay + context)
    ↓
PREVENTIVE COMPACTION (save before compress, summarize old exchanges)
```

### Phase 4: Constraint Mapping

**Interactive Focus:** Confronting all ideas with hard reality — technical constraints, skill gaps, scope, economics.

**9 Constraints Identified and Mitigated:**

1. **Not a Mobile Dev** → React Native/Expo or PWA + AI agents as co-devs
2. **LLM Orchestration on Mobile** → Dual loop (80% no LLM) + pre-compute + model routing
3. **Dynamic UI Rendering** → 30-50 pre-compiled components + JSON config (no code gen in V1)
4. **Data Security** → Keychain + encryption at rest + biometric + local-first + secrets never in SOUL
5. **Memory Complexity** → SQLite + sqlite-vec for V1 (graph in V2), OpenClaw proves this works
6. **Connector Ecosystem** → HTTP generic covers 80% + 5-7 dedicated connectors for V1
7. **Non-Technical Onboarding** → Agent IS the interface, zero jargon, SOUL stores user's tech level
8. **V1 Economics** → BYOK + local-first = near-zero infra cost, marketplace = GitHub repo
9. **MVP Scope** → Tight but sufficient: onboarding + chat + 15-20 components + memory + 5-7 connectors + BYOK

### Differentiation from OpenClaw

| Dimension | OpenClaw | Self |
|---|---|---|
| Starting point | Pre-configured AI assistant | Empty shell — builds from nothing |
| Customization | Skills and config | The ENTIRE app — UI, logic, features |
| Interface | Chat via existing apps (Discord, WhatsApp) | Native mobile app with visual bricks |
| Result | Similar assistants with different skills | 1M users = 1M completely different apps |
| Self-improvement | Agent learns and adapts | The app itself evolves — UI, layout, features |
| Memory | Markdown + SQLite (file-first) | Similar foundation + anti-repetition pipeline |
| Architecture | Single Node.js gateway | Mobile-first with offline support + dual performance loop |
