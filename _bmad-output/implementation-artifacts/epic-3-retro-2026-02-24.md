## What Went Well

1. **Product thesis validated end-to-end** — User describes a need in natural language → agent creates a native module → renders on phone. Confirmed on real Android device with weather metric (Météo Göteborg: 3°C) and news list (News Tech with articles + timestamps).

2. **3-layer architecture (Shell/Bridge/SDUI) proved its value** — By Story 3.4, the mobile side required ZERO code changes. `moduleSync.ts`, `chatSync.ts`, `wsClient.ts` — all infrastructure from prior stories "just worked."

3. **Registry pattern as force multiplier** — One line to add a new primitive, entire pipeline works. Grew from 3 to 7 registered types with zero breakage. Map-based registry (migrated from Record for prototype pollution safety in code review).

4. **TDD maintained with zero regressions** — Test counts grew steadily across all 4 stories. Red-green-refactor respected throughout. No regression between stories.

5. **Code reviews caught real security/quality issues** — Prototype pollution fix (3.1), overflow:hidden (3.2), scoped extractPrimitiveProps (3.3), import consolidation (3.3). Reviews are an essential safety net.

6. **Epic 1 retro lessons applied successfully** — snake_case/camelCase convention documented and followed: zero bugs related to case conversion in Epic 3 (was #1 bug source in Epic 1 with 3/7 stories affected). Async-only Python rule followed. Per-request DB connections pattern applied.

---

## What Didn't Go Well

1. **Mobile testing workflow extremely cumbersome** — 6 manual steps to get visual feedback: take screenshot on phone → send via Messenger → open on Mac → download → send to dev agent in terminal. Major velocity killer.

2. **Dev agent is blind** — Cannot see the actual render output of the code it writes. Relies only on unit tests that verify JSX structure, not visual rendering. The empty card bug in 3.4 (missing SDUI content fields in prompt) would have been caught instantly with visual feedback.

3. **26 pre-existing test failures (ChatInput SafeAreaProvider)** — Noise in the test runner at every run. Erodes confidence and makes it hard to distinguish real regressions. Plus `test_module_schema.py` import error on backend.

4. **Test count reporting inconsistent** — Story 3.1 claimed 95 tests (actual: 186), Story 3.3 claimed 779 (actual: 840), Story 3.4 claimed 612 backend (actual: 678). Self-reported metrics are unreliable.

5. **11 LOW review follow-ups accumulated** — Documented in stories but never resolved. Items span `accessibleRole` type safety, array index as React key, hardcoded strings, extractPrimitiveProps extra fields, redundant direction logic, JSON fence regex strictness.

6. **Post-implementation fix needed in 3.4** — Prompt engineering produced modules that rendered empty cards. Missing SDUI display content fields (value/label/unit for metric, items for list). Fixed after manual E2E testing caught it.

7. **No clear LLM model doctrine** — Story 3.3 used `claude-sonnet-4-6` while others used `claude-opus-4-6`. No documented rationale for model selection.

---

## Key Insights

1. **Visual feedback autonomy is a prerequisite, not a nice-to-have** — The dev agent must be able to launch the app, capture screenshots, read logs, and analyze results without human intervention. This is a blocker for the next wave.

2. **Well-laid infrastructure compounds exponentially** — Stories 3.1-3.3 built the SDUI pipeline; Story 3.4 benefited massively (zero mobile changes). Architecture investment pays off in later stories.

3. **Code reviews are a critical safety net** — 3 out of 4 stories had MEDIUM+ issues caught in review (prototype pollution, scoped props, import patterns). Must remain mandatory.

4. **Retro commitments work when executed** — Epic 1's #1 bug source (snake_case/camelCase) was completely eliminated in Epic 3 through documentation.

5. **Unit tests are necessary but not sufficient for UI** — Tests verify structure but not visual output. The prompt bug in 3.4 passed all tests but produced empty cards visually.
